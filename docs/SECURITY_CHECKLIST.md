# DataBridge Studio — 보안 체크리스트

> 금융권 / 공공 도입 시 감사 대응용.
> 운영 전 모든 항목 확인 필수. 확인자·확인일 기록 권장.

---

## A. 인증 · 권한

- [ ] **A1.** 기본 admin 계정 비밀번호 변경 완료
- [ ] **A2.** 모든 운영자 계정에 개별 ID 부여 (공용 계정 금지)
- [ ] **A3.** 역할 분리 확인: admin / operator / viewer 구분
- [ ] **A4.** 세션 타임아웃 설정 (기본 30분, 금융권 권장 15분)
- [ ] **A5.** 비밀번호 정책: 최소 12자, 영문+숫자+특수문자 조합
- [ ] **A6.** 로그인 실패 5회 후 계정 잠금 (현재 정책 확인)
- [ ] **A7.** (권장) 2FA 도입 계획 수립

## B. 통신 보안

- [ ] **B1.** 모든 웹 트래픽 HTTPS 강제 (TLS 1.2 이상, 1.3 권장)
- [ ] **B2.** 인증서 만료 60일 전 알림 (cron + 메일)
- [ ] **B3.** HSTS 헤더 설정 (`Strict-Transport-Security: max-age=31536000`)
- [ ] **B4.** HTTP → HTTPS 301 리다이렉트
- [ ] **B5.** 관리자 UI 접근 IP 화이트리스트 (사내망 한정)

## C. API 보안

- [ ] **C1.** `/metrics` 엔드포인트 방화벽 제한 (Prometheus 서버 IP 만)
- [ ] **C2.** `/health` 엔드포인트 내부망만 노출
- [ ] **C3.** `/api/v1/audit/*` — admin 권한 필수 확인
- [ ] **C4.** CORS 화이트리스트 운영 도메인 한정
- [ ] **C5.** API rate limiting 적용 (nginx limit_req 또는 백엔드 미들웨어)

## D. 저장 데이터 보안

- [ ] **D1.** 프로파일 비밀번호 Fernet 암호화 확인
  ```bash
  sqlite3 backend/data/databridge.db "SELECT value FROM store_profiles LIMIT 1"
  # 출력에 "password":"enc::..." 형태가 나와야 함 (평문 금지)
  ```
- [ ] **D2.** Anthropic API 키 암호화 저장 확인
- [ ] **D3.** 암호화 키 (Fernet key) 관리 정책 수립
  - 위치: `ENCRYPTION_KEY` 환경 변수 또는 별도 secret 저장소
  - 로테이션 주기 결정 (권장: 연 1회 이상)
  - 키 변경 절차 문서화
- [ ] **D4.** DB 파일 OS 권한 제한 (`chmod 640 databridge.db`, 소유자만 읽기)
- [ ] **D5.** 백업 파일 암호화 여부 확인 (민감 데이터 포함)

## E. 로깅 및 감사

- [ ] **E1.** 전역 PII 마스킹 필터 동작 확인 (v10 #21)
  ```bash
  # 로그 샘플링 → 주민번호/카드번호 패턴 grep 후 결과 없어야 함
  tail -n 1000 backend/logs/databridge.log | grep -E '[0-9]{6}-[0-9]{7}'
  ```
- [ ] **E2.** 감사 로그 해시 체인 무결성 — 월 1회 `/api/v1/audit/verify` 호출
- [ ] **E3.** 감사 로그 보존 기간 정책 수립 (금융권 최소 5년)
- [ ] **E4.** 감사 로그 외부 SIEM 연동 여부 결정
- [ ] **E5.** 로그 파일 접근 권한 제한 (운영자만 read)
- [ ] **E6.** 로그 파일 로테이션 (logrotate, 일 단위, 최근 30일 유지)

## F. 의존성 및 취약점

- [ ] **F1.** `requirements.txt` 버전 고정 (== pinning) 확인
- [ ] **F2.** Python 패키지 취약점 스캔
  ```bash
  pip install pip-audit
  pip-audit -r requirements.txt
  ```
- [ ] **F3.** npm 패키지 취약점 스캔
  ```bash
  cd frontend && npm audit --production
  ```
- [ ] **F4.** 정기 스캔 cron (월 1회) 등록
- [ ] **F5.** Critical/High 취약점 발견 시 대응 SLA 정의 (권장: 7일 이내 패치)

## G. DB 접속 (소스·타겟 DB)

- [ ] **G1.** 이관 작업용 DB 계정은 **최소 권한** 원칙
  - 소스 DB: SELECT 권한만 (메타 조회 + 데이터 읽기)
  - 타겟 DB: 대상 스키마 DDL/DML 권한만, 시스템 DB 접근 불가
- [ ] **G2.** 이관 완료 후 임시 계정 즉시 삭제
- [ ] **G3.** 방화벽: 서버 → DB 방향 포트만 허용
- [ ] **G4.** DB 서버 IP 고정 (동적 IP 금지)
- [ ] **G5.** 감사 요청 시 이관 계정의 DB 활동 로그 추출 절차

## H. 백업 및 복구

- [ ] **H1.** 일일 백업 cron 동작 확인 (실제 복구 테스트 완료)
- [ ] **H2.** 백업 보관 기간 정책 (권장: 30일 로컬 + 90일 원격)
- [ ] **H3.** 복구 시나리오 문서화 (RTO / RPO 정의)
- [ ] **H4.** 백업 파일 원격 사이트 보관 (재해 복구)
